## Analysis


The presence of XML in the request combined with the hint **“XML external entity injection”** suggests that the application may be vulnerable to **XML External Entity Injection (XXE)**.

### What is happening?

The server processes XML input using an XML parser. If external entities are enabled, an attacker can define a malicious entity that references a local file on the server.

When the XML parser processes the entity, it replaces the entity reference with the contents of that file.

### What is the vulnerability / trick?

The trick is to inject a **malicious XML DOCTYPE declaration** that defines an external entity pointing to `/etc/passwd`.

When the server parses the XML, the entity will resolve and return the contents of the file.

Example payload inspired by documentation from **PortSwigger**  https://portswigger.net/web-security/xxe.

```xml
<?xml version="1.0" encoding="UTF-8"?> 
<!DOCTYPE foo [ <!ENTITY xxe SYSTEM "file:///etc/passwd"> ]> 
  <stockCheck>
    <productId>&xxe;</productId>
  </stockCheck>
```

---

## Exploitation / Solution

1. Intercept the POST request using **Burp Suite**.
    
2. Modify the XML request body to include an **XXE payload**.
    
3. Define an external entity that reads `/etc/passwd`.
    
4. Reference the entity inside the XML element.
    
5. Send the modified request to the server.
    

### Exploit Payload

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE foo [
<!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<data>
<ID>&xxe;</ID>
</data>
```

![](7_2.png){: width="800" height="400" .shadow }

After sending the request, the response returned the contents of `/etc/passwd`.

## Key Idea

The server was vulnerable to **XML External Entity Injection (XXE)** because the XML parser allowed external entities.

By defining an entity pointing to `/etc/passwd`, the server resolved the entity and returned the file contents in the response.
